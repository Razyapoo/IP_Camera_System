# Документация новой архитектуры системы IP-камер

Данный документ описывает изменения, внесенные в ходе масштабного рефакторинга приложения, и подробно объясняет новую многофайловую объектно-ориентированную (ООП) архитектуру.

---

## Что было изменено: Основные улучшения

Оригинальный скрипт `video.py` был полностью переработан с процедурного стиля («спагетти-код» с глобальными переменными и синхронным вводом-выводом) на модульную архитектуру, соответствующую принципам SOLID и лучшим практикам программирования.

1. **Разделение ответственности (Single Responsibility Principle)**:
   Один монолитный файл был разделен на 5 специализированных модулей и 1 файл-запускальщик, каждый из которых отвечает строго за свою часть логики (конфигурация, сетевой поток, детекция движения, графический интерфейс, оркестрация).
2. **Многопоточное получение кадров (Asynchronous/Multi-threaded Streaming)**:
   Вместо синхронного последовательного чтения потоков в основном цикле, каждая активная камера теперь обслуживается собственным фоновым потоком-демоном (`StreamReader`). Кадры захватываются непрерывно в фоне. Это **полностью устранило фризы и зависания интерфейса**, если у одной из камер падает качество связи или увеличивается пинг.
3. **Легковесные сетевые проверки (Fast TCP Port Handshakes)**:
   Запуск ресурсоемких системных подпроцессов `ping` через ОС каждые 5 секунд был заменен на быстрые неблокирующие TCP-соединения (сокеты) на RTSP-порт камеры (554). Это снизило нагрузку на CPU практически до нуля и повысило точность проверки доступности видеосервиса.
4. **Улучшенная детекция движения**:
   Анализ движения теперь производится на **сырых исходных кадрах до наложения графики, часов и текста**. Это устранило ложные срабатывания детектора на изменения секунд в скринсейвере. Дополнительно применяется **размытие по Гауссу (Gaussian Blur)** для фильтрации цифрового шума матрицы.
5. **Атомарное обновление конфигурации**:
   Обновление файла `cameras_config.json` (например, при автоопределении бренда) теперь происходит атомарно через создание временного буферного файла и его системную замену (`os.replace`). Это исключает риск порчи или зануления конфигурации при внезапном отключении питания.
6. **Безопасная обработка исключений и логирование**:
   Конструкции глухого подавления ошибок `try-except: pass` заменены на перехват конкретных исключений с выводом трассировок через стандартную библиотеку `logging`.
7. **Обратная совместимость**:
   Файл `video.py` сохранен как обратная точка входа (launcher). Любые внешние скрипты, вызывающие `python video.py`, будут работать абсолютно прозрачно, запуская под капотом новую оптимизированную архитектуру.

## Новая архитектурная структура

Приложение разделено на следующие независимые модули в каталоге `D:\Data\IP_Camera_System\`:

```
D:\Data\IP_Camera_System\
├── cameras_config.json  # JSON Файл конфигурации
├── config.py            # Модуль управления настройками (ConfigManager)
├── camera.py            # Модуль работы с камерами и потоками (Camera, StreamReader)
├── motion.py            # Модуль детекции движения (MotionDetector)
├── ui.py                # Модуль графического интерфейса OpenCV (AppUI)
├── main.py              # Главный оркестратор приложения (CameraSystemApp)
└── video.py             # Совместимый файл-запускальщик (Legacy Launcher)
```

---

### Подробное описание модулей

#### 1. `config.py` (Класс `ConfigManager`)
Отвечает за загрузку настроек из JSON и предоставление их в виде удобных read-only свойств (properties).
*   **Метод `load_config`**: Загружает и парсит JSON, обрабатывая ошибки разметки.
*   **Метод `update_camera_brand_and_format`**: Безопасно и атомарно перезаписывает параметры бренда камеры и транспорта (TCP/UDP) в JSON, используя системные вызовы атомарной замены файлов.

#### 2. `camera.py` (Классы `Camera` и `StreamReader`)
Полностью инкапсулирует свойства конкретной камеры и управляет ее сетевым состоянием.
*   **Класс `Camera`**: Хранит учетные данные, IP, порт, каналы и текущие статусы (`is_online_cached`, `rtsp_working`, `paused`). Предоставляет метод `check_online_status()` на базе TCP сокетов и управляет жизненным циклом потока чтения кадров.
*   **Класс `StreamReader` (наследник `threading.Thread`)**: Фоновый поток, который в бесконечном цикле опрашивает `cv2.VideoCapture`. При паузе (`paused = True`) он автоматически освобождает сетевой буфер и «засыпает», снижая нагрузку на сеть.
*   **Метод `detect_brand_at_startup`**: Тестирует камеру по всем известным RTSP-шаблонам брендов (Dahua, Hikvision, Axis, Reolink, ONVIF) для автоопределения марки при первом запуске.

#### 3. `motion.py` (Класс `MotionDetector`)
Управляет алгоритмами обнаружения движения.
*   Хранит историю предыдущих кадров для каждой камеры индивидуально.
*   **Метод `detect_motion`**: Принимает словарь сырых кадров, переводит их в оттенки серого, сглаживает шумы фильтром Гаусса, рассчитывает разность и бинаризует изображение по порогу яркости, суммируя изменения. При превышении порога `motion_threshold` возвращает сигнал активности.

#### 4. `ui.py` (Класс `AppUI`)
Интерфейсный слой, абстрагирующий вызовы оконной системы OpenCV.
*   **Функция `get_screen_resolution`**: Кроссплатформенный детектор разрешения экрана (поддерживает Windows API, Linux `xrandr` / `xdpyinfo`, macOS `system_profiler` и Tkinter как резервный вариант).
*   **Методы отрисовки**:
    *   `show_startup_screen`: Красивый экран загрузки с Progress Bar и текстовым статусом для каждого этапа инициализации.
    *   `build_grid`: Сборка сетки $N \times M$ из отдельных изображений камер с автоматическим масштабированием под физический экран с сохранением пропорций (Aspect Ratio).
    *   `build_idle_screen`: Отрисовка скринсейвера (режима ожидания) со временем простоя и цифровыми часами.
*   **Метод `get_camera_index_from_coordinates`**: Математически рассчитывает, на ячейку какой именно камеры кликнул пользователь, учитывая физические размеры сетки на экране и черные поля (offset).

#### 5. `main.py` (Класс `CameraSystemApp`)
Главный мозг и оркестратор приложения, координирующий все компоненты вместе.
*   **Инициализация**: Создает экземпляры всех классов и связывает их через защищенные коллбеки (обработка кликов, переход в полноэкранный режим).
*   **Метод `run_startup_sequence`**: Запускает трехэтапную загрузку (1. Быстрый параллельный пинг сокетами; 2. Подключение RTSP и автоопределение брендов; 3. Старт фоновых видеопотоков).
*   **Метод `main_loop`**: Основной цикл, обрабатывающий переключения оконных состояний:
    *   **Режим сетки (Grid Mode)**: Получает последние кадры из потокобезопасных буферов камер, проверяет движение, обновляет таймеры активности и рендерит сетку.
    *   **Режим одиночной камеры (Single Camera Mode)**: Переводит все остальные камеры в режим паузы (выключая их фоновое чтение), переподключает выбранную камеру на полноэкранный поток высокого разрешения (`fullscreen_stream`) и выводит плавную картинку на весь экран.
    *   **Режим ожидания (Idle Mode)**: Гасит видеопотоки при отсутствии движения и активности клавиатуры, запуская скринсейвер.

## Сравнительный анализ архитектур

| Характеристика | Старая архитектура (`video.py` до рефакторинга) | Новая архитектура (После рефакторинга) |
| :--- | :--- | :--- |
| **Парадигма** | Линейное процедурное программирование | Объектно-ориентированное проектирование (ООП), SOLID |
| **Управление состоянием** | Десятки глобальных переменных и параллельных списков | Локальные свойства объектов внутри классов `Camera`, `AppUI` |
| **Чтение видеопотоков** | Синхронно в главном потоке интерфейса (вызывало зависания) | Независимые фоновые потоки `StreamReader` для каждой камеры |
| **Проверка доступности** | Запуск тяжелых системных утилит `ping` каждые 5 сек | Быстрые неблокирующие TCP-сокеты на RTSP-порт 554 |
| **Детекция движения** | Разность пикселей готовой сетки (реагировала на часы и текст) | Анализ чистых кадров до наложения графики + размытие Гаусса |
| **Запись конфигурации** | Обычная перезапись файла JSON (риск порчи структуры) | Безопасная атомарная замена файлов (`os.replace`) |
| **Логирование** | Пустые конструкции `try-except: pass` (баги скрывались) | Использование стандартного логгера `logging` |
